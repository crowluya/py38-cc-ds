# 项目宪法（Constitution）

**版本**: 1.0  
**日期**: 2024-12-19  
**状态**: 不可协商（NON-NEGOTIABLE）

## 前言

本文档定义了项目的工程原则和不可协商的红线。所有设计决策、代码实现、技术选型都必须遵循这些原则。违反宪法的方案或实现应被判定为"违宪"，必须修正。

## 一、简单性原则（Simplicity First）

### 1.1 YAGNI（You Aren't Gonna Need It）

- **原则**：只实现 `spec.md` 明确要求的功能
- **禁止**：为"未来可能的需求"提前实现
- **禁止**：过度抽象和设计模式堆砌
- **要求**：每个功能都必须有明确的用户故事或需求依据

### 1.2 标准库优先

- **原则**：优先使用 Python 标准库
- **要求**：引入第三方库必须有充分理由
- **禁止**：为了"看起来专业"而引入不必要的依赖
- **例外**：核心功能依赖（如 `click`, `rich`, `openai`）是必需的

### 1.3 反过度工程

- **禁止**：复杂的设计模式（除非明确需要）
- **禁止**：多层抽象和包装（除非有实际价值）
- **要求**：优先使用简单的函数和数据结构
- **要求**：代码应该让新手也能理解

## 二、测试驱动开发（TDD）不可协商

### 2.1 Red-Green-Refactor 循环

- **强制顺序**：先写测试（Red）→ 再写实现（Green）→ 最后重构（Refactor）
- **禁止**：同时写测试和实现
- **禁止**：先写实现再补测试
- **原因**：防止 AI 的"自洽幻觉"（写错实现，再写配合错误实现的测试）

### 2.2 表格驱动测试优先

- **要求**：核心逻辑必须使用表格驱动测试
- **要求**：覆盖正常情况、边界情况、错误情况
- **禁止**：只测试"happy path"

### 2.3 测试覆盖率要求

- **要求**：核心功能测试覆盖率 ≥ 70%
- **要求**：所有公共 API 必须有测试
- **要求**：关键路径必须有集成测试

### 2.4 集成优先于 Mock

- **原则**：优先使用真实依赖或 fake server
- **禁止**：过度 Mock（除非必要）
- **要求**：外部依赖通过依赖注入，便于测试

## 三、明确性原则（Explicitness）

### 3.1 错误处理必须显式

- **禁止**：使用 `_` 丢弃错误（Python 中 `_ = func()` 忽略返回值）
- **要求**：所有错误必须显式处理
- **要求**：错误包装使用 `%w` 格式（Python 中使用 `%w` 或 `from ... import ...`）
- **要求**：关键操作必须有错误处理和日志

### 3.2 依赖必须显式注入

- **禁止**：全局状态和单例模式（除非有充分理由）
- **要求**：依赖通过构造函数或参数传递
- **要求**：便于测试和替换实现

### 3.3 类型提示必须明确

- **要求**：所有公共 API 必须有类型提示
- **要求**：使用 `from typing import List, Dict, Optional, Union, Tuple`
- **要求**：Python 3.8.10 兼容的类型注解（使用字符串类型或 `from __future__ import annotations`）

## 四、单一职责与接口哲学

### 4.1 单一职责原则

- **要求**：每个模块、类、函数只做一件事
- **要求**：包/文件/函数职责高度内聚
- **禁止**："上帝类"和"万能函数"

### 4.2 接口设计

- **要求**：小接口、接口隔离
- **禁止**："上帝接口"（包含所有方法的大接口）
- **要求**：接口应该便于测试和替换实现

### 4.3 模块边界清晰

- **要求**：模块之间依赖关系清晰
- **要求**：避免循环依赖
- **要求**：核心模块不依赖扩展模块

## 五、Python 3.8.10 严格兼容

### 5.1 版本锁定

- **要求**：所有代码必须在 Python 3.8.10 上运行
- **禁止**：使用 Python 3.9+ 的特性
- **禁止**：使用 `:=` 赋值表达式（3.8.10 早期版本可能有 bug）
- **禁止**：使用 `f"{var=}"` 调试语法（3.8.10 可能不支持）

### 5.2 类型提示兼容

- **要求**：使用 `from __future__ import annotations` 或字符串类型注解
- **要求**：显式导入类型（`from typing import ...`）
- **禁止**：使用 Python 3.9+ 的类型特性

### 5.3 异步使用

- **要求**：使用 `asyncio.run()` 而非直接 `await`
- **要求**：谨慎使用异步，优先使用同步 `requests`

## 六、Windows 7 兼容性

### 6.1 路径处理

- **要求**：使用 `pathlib.Path` 统一处理路径
- **禁止**：硬编码路径分隔符（`/` 或 `\`）
- **要求**：用户目录使用 `pathlib.Path.home()` 或 `os.path.expanduser("~")`

### 6.2 命令执行

- **要求**：Windows 7 使用 PowerShell（`powershell.exe -Command`）
- **要求**：使用 `subprocess` 模块而非 `os.system()`
- **要求**：正确处理编码（UTF-8）
- **注意**：PowerShell 2.0 功能有限，使用兼容的命令和语法

### 6.3 编码处理

- **要求**：所有文件操作显式指定 `encoding='utf-8'`
- **要求**：命令输出正确处理编码（UTF-8）
- **禁止**：依赖系统默认编码

## 七、内网环境适配

### 7.1 依赖管理

- **要求**：提供 `requirements.txt` 和 `requirements-frozen.txt`
- **要求**：支持离线安装（wheel 文件）
- **要求**：文档说明内网安装方法

### 7.2 证书验证

- **要求**：支持自签名证书（`verify=False` 或自定义 CA）
- **要求**：证书配置可配置（不硬编码）
- **要求**：默认验证，但允许关闭（内网环境）

## 八、安全原则

### 8.1 最小权限

- **要求**：默认最小权限（只读顾问）
- **要求**：危险操作必须审批
- **要求**：文件访问、命令执行、网络访问都有权限规则

### 8.2 审批机制

- **要求**：写文件、执行命令、网络访问需要审批（ask 模式）
- **要求**：使用 `questionary` 做安全确认
- **要求**：审批历史可审计

### 8.3 检查点回滚

- **要求**：支持会话级检查点
- **要求**：支持 `/rewind` 回滚
- **要求**：检查点包含代码状态和对话状态

## 九、代码质量

### 9.1 代码审查

- **要求**：所有关键变更必须审查
- **要求**：使用 `/review` 命令进行代码审查
- **要求**：审查报告结构化输出

### 9.2 文档要求

- **要求**：所有公共 API 必须有文档字符串
- **要求**：复杂逻辑必须有注释
- **要求**：README 和使用文档完整

### 9.3 错误处理

- **要求**：错误信息清晰、可操作
- **要求**：关键错误有日志记录
- **要求**：错误处理不静默失败

## 十、规范驱动开发（SDD）

### 10.1 规范是真理之源

- **要求**：`spec.md` 是唯一权威
- **要求**：所有实现必须符合 spec
- **要求**：意图偏差修正 spec，实现偏差修正代码

### 10.2 人工审查

- **要求**：`plan.md` 必须人工审查
- **要求**：`tasks.md` 必须人工审查
- **要求**：不审查就进入实现，返工代价巨大

### 10.3 任务粒度

- **要求**：任务必须原子化（一个文件/一个函数/一个结构体）
- **要求**：任务依赖关系明确
- **要求**：可并行任务标记 `[P]`

## 十一、合宪性检查清单

在生成 `plan.md` 时，必须逐条对照宪法进行自检：

- [ ] 简单性门禁
  - [ ] 是否标准库优先？
  - [ ] 是否避免复杂抽象？
  - [ ] 是否只实现 spec 要求的功能？

- [ ] TDD 门禁
  - [ ] 计划中是否先写测试？
  - [ ] 是否使用表格驱动测试？
  - [ ] 是否覆盖多种场景？

- [ ] 明确性门禁
  - [ ] 错误处理是否显式？
  - [ ] 依赖是否显式注入？
  - [ ] 类型提示是否明确？

- [ ] 兼容性门禁
  - [ ] 是否 Python 3.8.10 兼容？
  - [ ] 是否 Windows 7 兼容？
  - [ ] 是否内网环境适配？

- [ ] 安全门禁
  - [ ] 是否最小权限？
  - [ ] 是否审批机制？
  - [ ] 是否检查点支持？

## 十二、违宪处理

如果发现方案或实现违反宪法：

1. **立即停止**：不继续实现
2. **修正方案**：回到 plan.md 或 spec.md 修正
3. **重新审查**：修正后重新审查
4. **记录原因**：在 plan.md 中说明修正原因
